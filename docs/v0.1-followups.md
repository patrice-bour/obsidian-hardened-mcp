## From M6 review (18550fe)

### M6-01 — HMAC field separator can theoretically collide
- **Severity**: Minor (defense-in-depth)
- **Status**: open
- **Refs**: `security/confirm.py:233-243`, `domain/vault_path.py:101-117`
- The HMAC message uses `\x1e` (ASCII record separator) between fields;
  `VaultPath.from_user` rejects only `\x00`, so a target whose relative
  path contains a literal `\x1e` could in theory yield the same HMAC
  message as a different (op, target, payload_hash) triple. The
  consume-side equality checks (`stored.target != expected_target`,
  `compare_digest(payload_hash)`) defend against this in practice — an
  attacker would still need to submit the exact stored values. Fix
  options: length-prefix each field (`struct.pack("!I", len(field))`),
  or extend `VaultPath` to reject ASCII control chars.

### M6-02 — Snapshot path containment is sandboxed by VaultPath, not asserted
- **Severity**: Minor (defense-in-depth)
- **Status**: open
- **Refs**: `fs/snapshot.py:66`
- The destination is `snapshot_root / snapshot_id / Path(*vp.relative.parts)`.
  Today VaultPath guarantees `vp.relative.parts` is non-empty and
  traversal-free. Add an explicit `assert dest.resolve().is_relative_to(
  snapshot_root.resolve())` after building the destination so a future
  bypass upstream cannot smuggle a snapshot outside `.ohmcp-trash/`.

### M6-03 — Backlink scan post-rename re-scan supersedes the src→dest remap
- **Severity**: Minor (dead code)
- **Status**: open
- **Refs**: `tools/destructive.py:660-684` (rename), `tools/destructive.py:945-963` (move)
- After `os.replace` the source no longer exists at its old relative
  path; the post-phase-2 re-scan therefore never returns
  `src_relative` in its candidate list, making the
  `target_rel = dest_relative if rel == src_relative else rel` remap
  inside `_rewrite_backlinks_phase2` unreachable. Pick one strategy:
  either drop the post-rename re-scan and trust the phase-1 candidates
  (and keep the remap), or drop the remap (and document that
  self-references in the source itself are rewritten only because the
  post-rename re-scan picks them up at the new path).

### M6-04 — Lazy registry init is not thread-safe
- **Severity**: Minor (single-writer model documented)
- **Status**: open
- **Refs**: `server.py:87-93`
- Two concurrent destructive calls could both observe an unset slot
  and both attempt `load_or_bootstrap_secret`; the second one's
  `os.open(O_CREAT|O_EXCL)` would raise `FileExistsError` (not caught).
  Fix: wrap the lazy bootstrap in a `threading.Lock`, or replace the
  closure cell with a `functools.lru_cache(maxsize=1)` wrapper.
  Acceptable for v0.1 because the project documents a single-writer
  assumption (CLAUDE.md §9), but trivially fixable for v0.2.

### M6-05 — Document that consumed-then-failed-snapshot tokens are unrecoverable
- **Severity**: Minor (UX / docs)
- **Status**: open
- **Refs**: `tools/destructive.py` (delete/rename/move phase-2 snapshot fail), `docs/security-model.md`
- If `snapshot_for_destruction` fails after the token was consumed,
  the audit records the failure but the token is gone — the user must
  call phase 1 again. Update the tool descriptions and security-model
  to make this explicit.

### M6-06 — `_verify_hmac` over-pads base64 input
- **Severity**: Minor (style / portability)
- **Status**: open
- **Refs**: `security/confirm.py:248`
- We always append `==` regardless of the canonical token length
  (always 86 chars, so adding `==` happens to make it valid). The
  defensive check on `len(raw)` already catches most degenerate input.
  Tighten by either computing `(-len(token_str)) % 4` for the correct
  pad count, or rejecting any token whose length is not exactly 86.

### M6-07 — Mode check `mode != 0o600` rejects more-restrictive modes
- **Severity**: Minor (debatable)
- **Status**: open
- **Refs**: `security/confirm.py:271-275`
- A user who tightens the secret to `0o400` (read-only owner) is
  refused even though the mode is strictly safer than `0o600`. Decide:
  document the choice ("we want read+write so we can rotate"), or
  accept any mode `<= 0o600` (no group/other bits set).

### M6-08 — `OperationName` Literal includes unimplemented `"batch"`
- **Severity**: Optional
- **Status**: open
- **Refs**: `security/confirm.py:42`
- Brief reserved `batch` for future multi-op tokens; M6 ships none.
  Either remove or comment as a roadmap reservation.

### M6-09 — `_FIELD_SEP` comment claims "won't appear in our fields" but the invariant isn't enforced
- **Severity**: Optional
- **Status**: open
- **Refs**: `security/confirm.py:36`
- Same root as M6-01; tighten the comment or the contract.

### M6-10 — Snapshot id stress test
- **Severity**: Optional
- **Status**: open
- **Refs**: `tests/unit/test_fs_snapshot.py:87`
- `test_two_rapid_calls_yield_different_ids` does 5 calls. Bump to
  100+ to catch any future regression in `_new_snapshot_id` (current
  4-byte hex suffix has a 1-in-4-billion collision probability per
  pair, so 100 calls is safely below detection threshold).

### M6-11 — 2-phase HMAC does not stop a coherently-hallucinating LLM
- **Severity**: Major (positioning / honesty)
- **Status**: open (doc-honesty pass shipped pre-flip; real fix deferred)
- **Refs**: `src/obsidian_hardened_mcp/security/confirm.py`,
  `docs/security-model.md` § "LLM-driven destructive ops"
- The 2-phase HMAC mechanism prevents a single hallucinated tool call
  from mutating, prevents token forge without the secret, prevents
  cross-target reuse, and prevents replay past TTL. It does **not**
  prevent an LLM that hallucinates phase 1, reads the returned token
  from its own context, then calls phase 2 with that token — both
  phases complete legitimately as far as the registry is concerned.
  Same applies to a prompt-injection-driven agent that walks the LLM
  through both phases.
- The real defence requires moving the second factor **out of the
  LLM's context** entirely. The MCP spec already supports this via
  `Context.elicit()` — servers can request structured input from the
  *user* (rendered by the client UI: Claude Desktop, Claude Code) at
  any point in the request lifecycle. An LLM cannot fabricate the
  elicitation response because it routes through the client, not the
  model.
- v0.2 plan: wrap `tools/destructive.py` phase-2 entry points so that
  before consuming the HMAC token, the server calls
  `ctx.elicit("Confirm <op> on <target>?", schema=...)` and only
  proceeds on an affirmative response from the user. Soft-fail with a
  clear error code (`elicitation_unsupported`) for clients that don't
  implement elicitation, so existing automation paths can opt in
  knowingly.
- Coupled with the existing snapshot trash + audit log, this would
  give the system three layers of defence against hallucinated or
  injected destructive intent: cryptographic binding (single-shot
  prevention), out-of-band confirmation (live human gate), recovery
  + detection (post-incident).

### M7-04 — `execute_command` allow-list
- **Severity**: Minor (defense-in-depth)
- **Status**: open

  the surface to a known-safe set (e.g. only navigation/UI commands,
  no `editor:delete-...`). The shape: `commands: { allow: [...] }`
  or `deny: [...]`. Reject phase-1 if the command isn't in the
  allow-list; reject early with `INVALID_PATH`.

### M7-S3 — RestClient lifecycle in long-lived servers
- **Severity**: Minor
- **Status**: open
- **Refs**: `server.py:111-119`
- `create_server` builds `RestClient` (which holds an `httpx.Client`
  with a connection pool) eagerly when a token is set, and never
  calls `RestClient.close()`. For typical stdio MCP usage the
  process exits with the client and that's fine, but a long-lived
  embedding (FastMCP behind an SSE host, future) would benefit from
  a lifecycle hook. Track until FastMCP exposes a clean shutdown
  callback in the server registration API.

### M7-S4 — Audit `vault_path=""` sentinel for command tokens
- **Severity**: Minor (operational)
- **Status**: open
- **Refs**: `tools/destructive.py:execute_command` audit calls
- Command-bound destructive audits write `vault_path=""`. Downstream
  JSONL filters that do `vault_path.startswith(folder)` match the
  empty string. Two choices: (a) switch to a clearly non-path
  sentinel like `"<command>"`, or (b) extend `AuditEvent` with an
  optional `target_command: str | None` field (cleaner schema, more
  invasive). Decide before the v0.1 tag.

### M7-S6 — Triple mutex check on OperationToken target/target_command
- **Severity**: Optional (DRY)
- **Status**: open
- **Refs**: `security/confirm.py` (post_init + issue + consume)
- The "exactly one of target / target_command" invariant is enforced
  in three places. With `frozen=True, slots=True` the dataclass
  invariant cannot be violated post-construction; the registry's
